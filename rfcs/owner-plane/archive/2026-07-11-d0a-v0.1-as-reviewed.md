# D0-A: Core + Memory — Normative Specification

*v0.1 draft, 2026-07-11. First of the four gate specifications under the
frozen umbrella RFC (`~/agenda-owner-plane-rfc.md`, v3.1/D0). **Gate rule:
no durable P1 Memory data is written until this document is frozen** —
frozen means: reviewed, every decision below either ratified or amended,
and the §13 golden vectors implemented and green in required keyless CI.
Normative language: MUST/NEVER are protocol requirements; SHOULD is
strong guidance; "v1" marks choices with an explicit upgrade path.*

## 1. Encoding profile (canonical bytes)

All signed or hashed objects use **deterministic CBOR** — RFC 8949
§4.2.1 Core Deterministic Encoding — further restricted:

- **E1.** Integers in shortest form; **no floats anywhere** in signed
  content; no bignums.
- **E2.** Definite-length encodings only (no indefinite strings, arrays,
  or maps).
- **E3.** Map keys are **UTF-8 text strings** in short `snake_case`;
  keys sorted by their canonical encoded bytes (length-first, then
  lexicographic — the RFC 8949 rule).
- **E4.** **No CBOR tags, no `null`, no `undefined`.** An absent value
  is an omitted key; presence of a key with a default value is a
  distinct, invalid encoding (validators MUST reject non-minimal
  encodings: unknown keys in signed objects, out-of-order keys,
  non-shortest ints).
- **E5.** Hashes, keys, IDs are **byte strings** (not hex text) inside
  CBOR; enums are text strings from closed vocabularies; timestamps are
  unsigned integers (Unix milliseconds).
- **E6.** Unknown fields: signed objects are **closed** (E4). Version
  bumps (`v` field) introduce new shapes; a validator that does not know
  `v` MUST treat the object as unverifiable and, when the object is
  effectful, **fail closed** (RFC §14 "schema downgrade").

**Domain separation.** Every hash and signature is computed over a
domain-prefixed message:

```
msg(type, x)   = "intendant/" || type || "/v1" || 0x00 || canonical(x)
H_type(x)      = SHA-256(msg(type, x))
Sign_ed25519   = Ed25519(sk, msg(type, x))
Sign_p256      = ECDSA-P256(sk, SHA-256(msg(type, x))), low-S (§2)
```

Type tags (closed set, one per object family): `genesis`, `cert`,
`grant`, `op`, `body`, `frontier`, `receipt`, `drill`, `kek`, `item`.
A signature or hash computed under one tag NEVER validates under
another.

**Rationale (CBOR over JCS-JSON):** binary canonical form with a
first-class deterministic profile, mature Rust (`ciborium`/`minicbor`)
and browser implementations, no number-representation pitfalls. Text
keys over integer keys: self-describing dumps are worth bytes at diary
scale (millions of items, not billions — RFC §4.7).

## 2. Algorithm suite v1 (`suite-v1`)

One composite suite ID, recorded in the genesis descriptor (RFC §3.1);
algorithm agility rides per-object algorithm IDs inside the suite.

| Role | Algorithm | ID | Notes |
|---|---|---|---|
| Hash | SHA-256 | (implied by suite) | matches vault/fingerprint precedent |
| Signature (daemon/native/recovery) | Ed25519 | `ed25519` | 64-byte sig |
| Signature (browser-held keys) | ECDSA P-256 w/ SHA-256 | `p256` | raw `r‖s` 64 bytes, **low-S mandatory** |
| KEM/recipient wrap | HPKE (RFC 9180) base mode: DHKEM(P-256, HKDF-SHA256) + HKDF-SHA256 + AES-256-GCM | `hpke-p256-v1` | components match shipped vault construction; HPKE itself is new work (RFC §4.9) |
| Content AEAD | AES-256-GCM | `a256gcm` | items §5, KEK wraps §5.4 |
| Recovery-key derivation | BIP39(24 words) → PBKDF2 seed → HKDF-SHA256 | §8.1 | deterministic phrase→key |

- **S1 (low-S).** ECDSA signatures MUST be emitted and accepted only in
  low-S form (`s ≤ n/2`). Rationale: operation identity is the hash of
  exact signed bytes (RFC §4.2); malleable S would let any relay
  (including a malicious Connect) transform a valid op into a
  byte-different valid duplicate — manufacturing false writer-fork
  evidence against an honest writer. Clients normalize WebCrypto output
  before emitting; validators reject high-S. A golden vector pins the
  rejection.
- **S2 (browser reality).** Browser-held signing keys are
  non-extractable WebCrypto P-256 (existing custody precedent). Ed25519
  verification in browsers uses WebCrypto where available with a pure-JS
  fallback — verification is not custody. Browser-held KEM keys are
  non-extractable P-256 ECDH used via `deriveBits` inside the HPKE
  key schedule (all HPKE primitives — ECDH, HKDF, AES-GCM — are native
  WebCrypto; the key schedule is library code).
- **S3 (weakest wrap).** Inherited from RFC §4.9: a zone remains
  classically exposed while any recipient wrap is classical; every wrap
  carries its KEM algorithm ID so X25519/hybrid ML-KEM are additions.
- **S4 (Rust side).** Pure-Rust implementations only (repo convention):
  `ring`/`ed25519-dalek`-class primitives + a pure-Rust HPKE (RustCrypto
  `hpke`). No OpenSSL.

## 3. Identifiers

| Identifier | Form | Derivation |
|---|---|---|
| `plane_id` | 32 bytes | `H_genesis(descriptor)` (§4.1) |
| `device_id` | 16 bytes random | minted at certificate issuance; stable across cert renewals |
| `zone_id`, `space_id` | ULID (16 bytes) | minted by control ops; public surfaces see opaque handles only (D0-B) |
| `writer_id` | 16 bytes | minted per writer generation (RFC §4.2); the control chain uses the reserved value §3.1 |
| `request_id` | 16 bytes random | unique per writer chain; duplicate with different bytes = fork evidence |
| `grant_id`, `revocation_id` | 16 bytes random | grant/cert lifecycle handles |
| claim / judgment / pin identity | 32 bytes | the emitting operation's `op_hash` — no separate ID space |

- **N1 (reserved control namespace).** Control operations use the
  reserved literals `tenant = "ctrl"`, `zone_id = ctrl-zone`,
  `space_id = ctrl-space`, `writer_id = ctrl-writer`, where the
  reserved values are the 16-byte constants
  `0x00…00 || "CTRLZONE"`-style fixed bytes pinned in the vectors.
  Rationale (umbrella erratum 1): **one envelope shape, one signing
  domain** — control ops are representable without a second header
  format; validation rules differ by tenant, encoding does not. The
  control feed is a single-writer chain v1 (§7.1).
- **N2 (space marker file).** The tracked repository marker (RFC §2.3)
  is named **`.intendant-space`**, TOML, exactly:

  ```toml
  version = 1
  space_id = "<ULID text form>"
  ```

  It is a binding *request*, never authority: the daemon binds it to a
  plane space only with owner/IAM confirmation and keeps the local
  binding map (space_id → checkout paths). Tests: copied-marker
  rejection; all worktrees of one repo resolve to one space identity.

## 4. Core objects and canonical encodings

Field lists below are normative; the golden vectors (§13) pin exact
bytes. All objects begin with `v: 1`.

### 4.1 GenesisDescriptor (`genesis`)

```
{ v: 1,
  suite: "suite-v1",
  root_sig_alg: "ed25519" | "p256",
  root_sig_pk: bytes,
  recovery_commitment: bytes(32),     # H_drill(recovery signing pubkey), §8
  governance: { v: 1, kind: "single-owner" },
  provenance: "trusted" | "hosted",   # RFC §3.5, carried forever
  created_ms: uint }
plane_id = H_genesis(descriptor)
```

- `root_sig_alg` is `p256` on the hosted lane (browser-held root) and
  `ed25519` on trusted/native genesis — both admissible; the ceiling
  derives from `provenance`, not the algorithm.
- The recovery authority appears only as a **hash commitment**: the
  recovery public key stays unrevealed until a recovery or drill
  operation reveals it (§8). Genesis without a recovery commitment is
  invalid — the independent authority is mandatory (RFC §3.1).
- `governance.kind = "single-owner"` is the only v1 value; the field is
  versioned for org/multi-party schemes (separate RFC).

### 4.2 DeviceCertificate (`cert`)

```
{ v: 1, plane_id, device_id,
  sig_alg, sig_pk, kem_alg: "hpke-p256-v1", kem_pk,
  class: "hosted-browser" | "owner-browser" | "native-app"
       | "daemon" | "mobile" | "other",
  evidence_hash: bytes(32),          # per-class evidence doc, RFC §3.2
  issued_admin_epoch: uint,
  expiry_deadline_ms: uint,          # acceptance-deadline semantics, §9
  revocation_id: bytes(16) }
cert_hash = H_cert(certificate)
signature: by the admin authority (proof arm GenesisAuthority or
AdminEpoch — §4.4); the signature and proof ride the enclosing
EnrollDevice control operation, not the certificate body.
```

Signing and KEM keys are distinct keys with distinct purposes; an
Ed25519 signing key is NEVER converted to X25519 (RFC §3.2). The
evidence document (hashed here) is class-specific: origin/bundle facts
for browser classes; locally verified artifact + explicit owner
attestation for native/daemon; platform attestation for mobile;
`other` is the explicit weaker class. Provenance means custody
ceremony, never software incorruptibility.

### 4.3 CapabilityGrant (`grant`)

```
{ v: 1, plane_id, grant_id,
  subject_cert: bytes(32),           # cert_hash
  writer_id?: bytes(16),             # present when the grant binds a writer chain
  tenants: [text],                   # ⊆ {"memory","agenda"}; "ctrl" is never grantable
  zones: [bytes(16)] | "*",
  spaces: [bytes(16)] | "*",
  ops: [text],                       # closed vocabulary per tenant, §10/§11
  kinds?: [text],                    # memory kinds constraint
  class_ceiling: "public"|"internal"|"private"|"sensitive",
  flows?: [ { from_zone, from_space, to_zone, to_space,
              kinds?, class_ceiling, expiry_deadline_ms } ],
  budget?: { max_ops: uint, max_bytes: uint },   # per capability_epoch
  online_lease: bool,                # high-impact: fresh-frontier required per use
  issued_admin_epoch: uint,
  capability_epoch: uint,
  expiry_deadline_ms?: uint }
grant_hash = H_grant(grant)
```

Grants are immutable; revocation is a control op naming `grant_id`.
Wildcards are the literal text `"*"` and match all *current and future*
resources of that axis — validators expand them at evaluation time,
never at issuance.

### 4.4 AuthorizationProof

The envelope's authorization slot (umbrella erratum 1) — a tagged
union, encoded as a CBOR map with discriminator `arm`:

```
{ arm: "dev",      cert: bytes(32), cap: bytes(32) }
{ arm: "genesis",  genesis: bytes(32) }               # root key implied by descriptor
{ arm: "admin",    epoch: uint, ctrl_frontier: bytes(32) }  # H_frontier of §4.6
{ arm: "recovery", repoch: uint, recovery_pk: bytes }  # revealed key, checked vs commitment
```

Arm admissibility is the §7.2 authorization matrix. The `admin` arm
binds the signer's view of the control frontier — a stale-root
detection input, not a freshness proof by itself (§7.4).

### 4.5 SignedOperation envelope (`op`, `body`)

One shape for tenant and control operations (RFC §4.1 as errata'd):

```
header {
  v: 1, tenant: "memory"|"agenda"|"ctrl",
  plane_id, zone_id, space_id,
  authored_crypto_epoch: uint,        # the ONLY epoch in signed bytes
  capability_epoch: uint,
  signer_alg, signer_key_id: bytes(32),   # H of signer pubkey
  writer_id, actor: { kind: "human"|"daemon"|"browser"|"agent-session"|"peer",
                      id: text, attested_by?: bytes(32) },
  authorization_proof: AuthorizationProof,
  request_id, writer_sequence: uint, previous_writer_hash: bytes(32),
  causal_references: [bytes(32)],
  created_hlc: [uint ms, uint logical],
  operation_type: text, operation_version: uint,
  body_hash: bytes(32) }              # H_body(body)

SignedOperation = { header, signature, body }
```

- **O1.** The signature covers `msg("op", header)` — the header only.
  Body integrity rides `body_hash` (`H_body` of the canonical body);
  this permits header-only validation and index/projection references
  to bodies. `body_hash` covers the **canonical plaintext body** and
  can never be re-decided (umbrella §4.0).
- **O2.** `op_hash = H_op({header, signature, body})` — the durable
  operation identity; replicas store these exact bytes and never
  verify-after-reserialize.
- **O3.** `authored_crypto_epoch` is the zone-KEK epoch the author
  encrypted under at write time. The mutable `key_wrap_epoch` lives in
  the item record (§5.3), never in signed bytes — zone-rotation erase
  rewraps without touching signatures.
- **O4.** The actor is a structured principal; supervised agent
  sessions hold no durable keys — their controller daemon signs and
  sets `attested_by` to its own cert hash (RFC §3.3). Free-form source
  strings are never authorization.
- **O5.** HLC is approximate chronology only; nothing in this document
  keys authority off it (RFC §4.2).

### 4.6 Frontier (`frontier`)

The canonical logical frontier — the type is D0-A because `AdminEpoch`
signs one (umbrella erratum 3):

```
{ v: 1, heads: { writer_id (bytes as map key) →
                 { seq: uint, op: bytes(32) } } }
H_frontier(frontier) = its 32-byte identity
```

Map keys sort per E3. D0-B adds Merkle compression and witness proofs
over this type; it never redefines it. Every surface that says
"frontier" means this structure; `ctrl_frontier` is this type over the
control chain (a single entry v1).

### 4.7 Receipts and lease proofs (`receipt`)

Deterministic expiry (§9) needs signed time-bearing statements with
pinned shapes:

```
StorageReceipt   { v:1, kind:"storage",   signer:"connect", key_id,
                   object: bytes(32), size: uint, ts_ms: uint }
ReplicaAck       { v:1, kind:"replica",   device: bytes(32),
                   frontier_hash: bytes(32), ts_ms: uint }
CheckpointWitness{ v:1, kind:"witness",   device: bytes(32),
                   checkpoint: bytes(32), ts_ms: uint }
```

Each is signed under `msg("receipt", x)` by its named signer. Signer
admissibility: plane devices always qualify; **Connect qualifies only
where root-signed zone policy sets `accept_connect_time = true`** — it
never silently becomes an authorization clock (RFC §4.5). Receipts
name their signer and their exact claim; they are durability/rollback
evidence per the two axes (RFC §4.4), never authority.

## 5. Item crypto (RFC §4.0/§4.7)

The **item is the AEAD unit**; segments (D0-B) only batch item records.

### 5.1 Item DEK and AEAD

- Every item (one SignedOperation's bytes) gets a fresh **32-byte
  random DEK**. **A DEK encrypts exactly once, ever** — re-proposal,
  compaction, and migration produce new items or move ciphertext
  verbatim; nothing re-encrypts under an old DEK.
- `item_ct = AES-256-GCM(dek, nonce, aad, plaintext = exact
  SignedOperation bytes)`.
- **Nonce**: 96-bit random, stored in the item record. (The single-use
  invariant would permit a zero nonce; a stored random nonce removes
  the foot-gun if any future path ever re-keys — defense in depth at
  zero cost.)
- **AAD** = `"intendant/item/v1" || 0x00 || plane_id || zone_id`.
  Cross-plane/zone ciphertext substitution fails AEAD before signature
  validation is ever reached.

### 5.2 Zone KEKs

- Per (zone, kek_epoch): a 32-byte KEK. Epoch 1 at zone creation;
  rotation mints epoch n+1 (revocation and erasure both ride this,
  RFC §4.5/§4.7).
- KEK distribution to enrolled recipients: HPKE base mode
  (`hpke-p256-v1`) to each recipient certificate's KEM key, with
  `info = "intendant/kek/v1" || 0x00 || plane_id || zone_id ||
  kek_epoch(uint64-be)`. Each wrap records its KEM algorithm ID (S3).

### 5.3 Item record (the mutable outer layer)

```
ItemRecord {
  v: 1, op_hash, ct: bytes, ct_hash: bytes(32),   # SHA-256(ct)
  nonce: bytes(12),
  key_wrap_epoch: uint,                # MUTABLE — current wrapping epoch
  wrapped_dek: { nonce: bytes(12), ct: bytes } }   # AES-256-GCM under KEK(key_wrap_epoch)
```

DEK wrap AAD = `"intendant/dekwrap/v1" || 0x00 || plane_id || zone_id
|| kek_epoch || ct_hash` — a wrapper is bound to its exact ciphertext
and epoch; wrapper substitution across items or epochs fails closed.
`ct_hash` is the item's content address (a separate outer identity
from `op_hash`, which only zone members can compute).

### 5.4 Zone-rotation erase and tombstones

Erase (decided, RFC §4.7) = rotate the zone KEK epoch; rewrap every
surviving item's DEK under the new KEK; **omit** erased items' wrappers;
checkpoint and GC all old segments/wrappers behind the fence; destroy
the old KEK on every cooperative replica and backup. The tombstone:

```
EraseTombstone { v:1, ct_hash, erase_op: bytes(32),  # the authorizing op
                 retired_epoch: uint }
```

No plaintext-derived hash appears in a tombstone (a plaintext hash
would let anyone holding a guess confirm it). Effective against
cooperative replicas/backups only — retained KEKs or plaintext stay
outside the guarantee (RFC §4.5); indexes/projections/checkpoints/
backups share the erasure boundary.

## 6. Local durable storage (RFC §4.2, §4.4)

### 6.1 Daemon log format

Append-only file per (plane, zone), plus one control log per plane:

```
record = len: u32-le || crc32c: u32-le || type: u8 || payload
types: 0x01 SignedOperation bytes   0x02 ItemRecord
       0x03 ReceiptRecord           0x04 FrontierCheckpoint
       0x05 OutboxMark              0x06 EraseTombstone
```

- **L1 (atomicity).** A record is committed when its full length is
  durable (`fsync` on the file, and on the directory for creation).
  Recovery truncates to the last record whose length and CRC both
  validate — torn tails are discarded, never repaired.
- **L2 (sequencing).** Writer-sequence allocation is durable before
  the op is emitted anywhere (the fsync'd allocation is the same append
  that stores the op). The outbox is the set of ops past the last
  acked frontier (`local-only` on the availability axis) — a durable
  replica + outbox, never a cache.

### 6.2 Browser mapping

IndexedDB object stores `ops`, `items`, `receipts`, `frontier`,
`meta`, written in single transactions per logical commit — the
platform's atomic durable transaction; there is no browser fsync to
promise (RFC §4.2).

- **L3 (writer lease).** One tab holds the writer lease via the Web
  Locks API (`navigator.locks`, exclusive, held for the tab's life).
  Only the lease holder allocates writer sequences.
- **L4 (writer generation).** `meta.writer_generation` increments on
  every restore-from-sync, import, or detected divergence; a new
  generation is a **new writer_id** (RFC §4.2's re-enrollment rule) —
  the old chain is never continued across uncertain state. Lease +
  generation together are the minimum two-writer rules P2a requires.
- **L5 (self-fork check).** Before writing after any gap (eviction
  suspected, restore, snapshot), reconcile the own frontier against a
  witnessed checkpoint where rollback protection is required; on
  detected self-fork, **stop writing as that writer** and surface (RFC
  §4.2).

## 7. Control reducer (RFC §3.1, §3.5)

### 7.1 The control chain

All control operations form **one single-writer chain** per plane
(`writer_id = ctrl-writer`, tenant `ctrl`). v1 governance is a single
owner: concurrent control ceremonies are not a supported workflow, so
a duplicate `(ctrl-writer, sequence)` **is** the competing-ceremony
case and fails closed (§7.4). Control operation types:

```
GenesisBundle      { descriptor, first_certs: [EnrollDevice…],
                     first_grants: [IssueGrant…] }   # sequence 1
EnrollDevice       { cert }               RevokeDevice   { revocation_id }
IssueGrant         { grant }              RevokeGrant    { grant_id }
ZoneCreate         { zone_id, name_hash } KekRotation    { zone_id, new_epoch,
                                                           wraps: [recipient wraps] }
CutoffSet          { writer_id, accepted_through: {seq, op} }
AbandonWriter      { writer_id, at: {seq, op} }
CheckpointCommit   { checkpoint_hash, covers_frontier: bytes(32), gc_fence }
AdminSuccession    { new_admin_pk, new_admin_alg, epoch }      # epoch = current+1
RecoverySuccession { new_admin_pk, new_admin_alg, epoch,
                     new_recovery_commitment }
DrillProof         { nonce: bytes(16) }   # §8.3
EraseAuthorize     { zone_id, ct_hashes: [bytes(32)] }
```

### 7.2 Authorization matrix (proof arm × operation)

| Operation | `genesis` | `admin(e)` | `recovery` | `dev` |
|---|---|---|---|---|
| GenesisBundle (seq 1 only) | ✅ | — | — | — |
| EnrollDevice / RevokeDevice | — | ✅ current e | — | — |
| IssueGrant / RevokeGrant | — | ✅ current e | — | — |
| ZoneCreate / KekRotation / CutoffSet / AbandonWriter / CheckpointCommit / EraseAuthorize | — | ✅ current e | — | — |
| AdminSuccession (to e+1) | — | ✅ signer = admin(e) | — | — |
| RecoverySuccession | — | — | ✅ | — |
| DrillProof | — | — | ✅ | — |
| Any tenant operation | — | — | — | ✅ |

- The `genesis` arm is valid **exactly once**, at control sequence 1.
  Epoch 1's admin key is the genesis root key.
- The `dev` arm NEVER authorizes control operations; `ctrl` is never a
  grantable tenant (§4.3).
- The `recovery` arm's signer is the revealed recovery key; validators
  check `H_drill(recovery_pk) == current recovery_commitment`.

### 7.3 Hosted ceiling invariant (RFC §3.5)

Reducer invariant, derived from genesis — never a policy document:

> While `genesis.provenance == hosted` and no `RecoverySuccession`
> whose new admin key was minted on a trusted client has been accepted:
> **(a)** no certificate with class other than `hosted-browser` may be
> enrolled; **(b)** no grant may carry `class_ceiling` above
> `internal`, any `flows`, or effect-approval operations; **(c)** the
> admissible control operations are exactly EnrollDevice
> (class `hosted-browser`), RevokeDevice (same class), and the tenant
> data ops those grants allow. Effects can be neither approved nor
> fired — a pure hosted-genesis plane is notes, tasks, and reminder
> drafts.

The hosted-to-trusted ceremony (RFC §3.5) is, in reducer terms, one
`RecoverySuccession` submitted from the trusted client: authorized by
the `recovery` arm (never by the hosted admin root — a
later-compromised hosted root must not self-succeed), carrying a fresh
native-held admin key and a fresh recovery commitment. Its acceptance
lifts (a)–(c) for newly enrolled certificates, retires the hosted
admin branch (§7.4), and the fresh root's first `EnrollDevice` is the
trusted client's certificate — enrollment is the ceremony's outcome.

### 7.4 Epochs, precedence, stale roots, replay

- **C1 (total order).** Admin epochs are consecutive integers. An
  `AdminSuccession` is valid only when signed by `admin(e)` with
  `epoch == e+1` and it extends the current control head.
- **C2 (control fork = fail closed).** Two control ops at the same
  `(ctrl-writer, sequence)` with different bytes — including two
  competing successions — freeze the control plane: no further control
  op on either branch is accepted, tenant writes continue under the
  last uncontested frontier, and the fork surfaces to the owner.
  **Only a `RecoverySuccession` resolves a control fork.**
- **C3 (recovery precedence).** A valid `RecoverySuccession` with
  `repoch == current_repoch + 1` supersedes **all** admin branches: it
  names the fork point it resolves (via `previous_writer_hash` +
  `causal_references` to the abandoned heads), sets
  `epoch = max(observed epochs) + 1`, installs the new recovery
  commitment, and retires every competing branch. Recovery epochs are
  consecutive integers; a competing pair of RecoverySuccessions at the
  same repoch is a **plane-compromise state**: everything freezes and
  resolution is out of band (the attacker holds the recovery secret —
  no protocol rule can adjudicate that).
- **C4 (stale root).** Control ops signed under `admin(e)` are invalid
  once `admin(e+1)` is accepted (control ops require the current
  epoch). Tenant ops authorized by grants issued at e remain valid per
  capability-epoch rules (§9) — grant validity is not implicitly tied
  to admin succession; revocation is explicit.
- **C5 (replay).** Control ops carry `request_id` and chain like any
  writer feed; a replayed op is byte-identical (idempotent, ignored)
  or a fork (C2). The `admin` arm's `ctrl_frontier` records the
  signer's view; validators use it for stale-root diagnostics, never
  as a freshness proof.

## 8. Recovery (RFC §3.1, §3.5; App C #3 partially resolved)

### 8.1 The independent recovery authority (v1)

- **R1.** The recovery authority is an **offline Ed25519 keypair
  deterministically derived from a 24-word BIP39 phrase** (256-bit
  entropy): `seed = PBKDF2-HMAC-SHA512(mnemonic, "mnemonic", 2048)`
  (standard BIP39, empty passphrase), then
  `sk = HKDF-SHA256(seed, info = "intendant/recovery/v1")[0..32]` as
  the Ed25519 seed. Deterministic derivation is the point: the phrase
  *is* the key, reconstructible on any trusted device.
- **R2.** v1 custodian: **the owner, offline** (written/printed
  phrase). Threshold custodian sets and social recovery are
  `governance_scheme` v2+ material (org/family RFC) — the commitment
  field does not change shape for them.
- **R3.** No online component ever stores the phrase or derived secret.
  At genesis the phrase is generated, displayed once, its public-key
  commitment written into the descriptor, and the secret material
  discarded. (Computing the commitment requires Ed25519 key *generation*
  at genesis — pure-library code on hosted genesis, since WebCrypto
  Ed25519 is not universal; acceptable because generation is momentary
  and the phrase, not the key object, is the custody.) Hosted genesis displays it inside the admitted hosted TCB
  (RFC §3.5) — which is exactly why re-root mints a **fresh** phrase on
  the trusted client (§7.3) and never re-enters the old one in hosted
  code.

### 8.2 Loss vs compromise (distinct procedures)

- **Loss** (admin key gone, recovery phrase held): `RecoverySuccession`
  from a trusted client — new admin key, new recovery commitment
  (mandatory rotation of **both**, RFC §3.1), old branches retired.
- **Compromise** (admin key attacker-held): same operation; C3's
  precedence makes it win over any attacker-signed admin branch, and
  the retired branch's certificates/grants are dead for control
  purposes from the fork point. Tenant ops the attacker wrote before
  the cutoff are handled by `CutoffSet` + quarantine (RFC §4.5) —
  evidence, not silently erased history.
- **Phrase loss** with working admin key: rotate the commitment via a
  fresh phrase — `RecoverySuccession` is not needed; this is
  `AdminSuccession`-adjacent but touches the recovery commitment, so
  it is a distinct op in v1.1 if demanded; v1 keeps it simple: generate
  a new plane-recovery phrase only through `RecoverySuccession`
  (requiring the old phrase) or accept that a lost phrase + healthy
  admin key means *migrating* to a re-rooted plane. **Flagged in §15 as
  the one recovery flow v1 deliberately does not smooth.**
- **Both lost**: unrecoverable by design; stated plainly in product
  copy.

### 8.3 Drill

`DrillProof { nonce }` — signed by the recovery key (revealing its
public key; harmless, custody is the secret). Validators check the
commitment and record a successful drill in the control history. The
drill proves the phrase still derives the committed key **without
rotating anything**. Drill cadence and UX are product policy; the
operation and its audit trail are protocol. Drills run on the trusted
lane only.

## 9. Deterministic expiry and budgets (RFC §4.5; App C #2 adopted)

- **X1 (acceptance deadline).** For ordinary capabilities and
  certificates, `expiry_deadline_ms` means: an operation authorized
  under them is *accepted at fold time* iff a qualifying receipt
  (§4.7) exists with `object == op_hash` (or a frontier/checkpoint
  covering the op) and `ts_ms ≤ expiry_deadline_ms`. Late-arriving ops
  without such a receipt land in quarantine for re-proposal — never
  silently folded, never silently dropped.
- **X2 (qualifying signers).** Plane devices always; Connect iff
  root-signed zone policy `accept_connect_time = true`.
- **X3 (no local clocks in the fold).** Deadline comparisons are
  receipt-timestamp vs deadline. Validator-local time appears in
  exactly one place: **online leases** — a grant with
  `online_lease = true` requires, per use, a control-frontier proof
  no older than `max_age_ms` (grant field, default 3600000) by the
  *executing service's* clock, with ±300s (`skew_ms = 300000`)
  allowance. Stale state fails closed absent an explicit bounded
  offline grant.
- **X4 (budgets).** `budget {max_ops, max_bytes}` is evaluated per
  `(writer_id, capability_epoch)` in writer-sequence order; the first
  op exceeding a bound and every subsequent op under that grant-epoch
  is invalid (deterministic — no shared-pool division offline,
  RFC §4.5).
- **X5 (strict vs lenient).** The strict-vs-lenient revocation-window
  choice is a root-signed, versioned **zone policy object**
  (`ZonePolicy { v, zone_id, strictness: "strict"|"lenient",
  accept_connect_time, … }`, installed by a control op) — never a
  local UX choice.
- **Freeze checkpoint (App C #2):** X1–X4 are normative now; the
  umbrella requires confirming the acceptance-deadline UX against real
  offline usage before D0-A freeze. The confirmation run is a §13 test
  scenario (offline writer across a deadline), not a redesign license.

## 10. Plane IAM — typed, fail-closed (RFC §6.2)

### 10.1 Decision input (one struct, no bare strings)

```
DecisionInput {
  principal { actor, cert_hash, capability: grant_hash,
              session_grant?: local evidence ref,     # daemon-local, audit only
              token_scope?: [ops] }                    # live session token, §8 RFC
  request   { tenant, op, plane_id, zone_id, space_id,
              kind?, status_filter?, classification?,
              flow? { from, to }, cost { ops: 1, bytes } }
  context   { ctrl_frontier, policy_id, policy_version, now_ms? }  # now_ms only for online leases
}
```

### 10.2 Evaluation (all five, in order, fail-closed)

```
effective = device capability (grant scope match: tenant ∧ zone ∧ space ∧ op ∧ kind)
          ∩ daemon-local session grant        (present, live)
          ∩ live session-token scope          (present, unrevoked, op ∈ scope)
          ∩ request resource constraints      (classification ≤ ceiling; flow ∈ flows;
                                               budget not exhausted (X4);
                                               expiry/lease satisfied (X1/X3))
          ∩ device provenance ceiling          (§7.3 hosted ceiling; class-based caps)
```

Any absent, expired, revoked, or non-matching component denies. Deny
reasons are a closed enum (pinned for audit + parity tests):
`no-grant, scope-tenant, scope-zone, scope-space, scope-op, scope-kind,
class-ceiling, no-flow, budget, expired, revoked, no-session,
no-token, token-scope, provenance-ceiling, stale-frontier, policy`.

- Agent/peer plane access **fails closed without an explicit plane
  grant** — the daemon-IAM root-compatible default for unscoped
  supervised sessions does NOT extend to plane operations (RFC §6.2).
- Sensitive-space reads/searches audit `(principal, scope, result
  IDs)` into the source zone or the owner-audit domain — never raw
  queries or content (RFC §6.2).

## 11. Memory tenant (RFC §6)

### 11.1 Operations (tenant `memory`, closed vocabulary)

`memory.search`, `memory.read`, `memory.evidence.read`,
`memory.propose`, `memory.assert`, `memory.curate`, `memory.export`,
`memory.admin`. Search returns bounded index entries; read returns
bodies; evidence dereference is separately authorized.

### 11.2 Bodies and judgments (operation_type → body shapes)

```
claim (op type "m.claim") body:
{ kind: "observation"|"decision"|"episode"|"procedure"|"preference",
  statement: text,
  sensitivity: "public"|"internal"|"private"|"sensitive",   # writer CLAIM
  observed_at_ms?, valid_from_ms?, valid_until_ms?, expires_at_ms?,
  provenance { session?, project?, model?, evidence: [bytes(32)] },
  supersedes?: [bytes(32)],        # ADVISORY lineage only
  labels?: [text] }

judgment (op type "m.judge") body:
{ verdict: "accept"|"dispute"|"retract"|"retire",
  target: bytes(32) }                                # claim op_hash
| { verdict: "supersede", target, replacement: bytes(32) }
| { verdict: "declassify", target, new_class }       # own capability
  — plus (all variants): policy: { id: text, version: uint },
    reason?, evidence?: [bytes(32)]
  (the judgment records the policy it was issued under — provenance;
   the fold's evaluation policy P is chosen at read time, §11.3/§11.4)

pin (op type "m.pin") body:
{ target: bytes(32), destination { space, role },
  expiry_ms?, token_budget?, provenance_floor?,
  accepted_under { judgment: bytes(32), policy_id, policy_version } }
unpin (op type "m.unpin") body: { target_pin: bytes(32) }
```

`memory.assert` = the claim op **plus** an accept judgment on it in the
same causal breath (the judgment's `causal_references` include the
claim), authorized under a named workflow policy — the trust
difference between propose and assert is replayable, never ambient.

### 11.3 Derived status — the fold

Inputs: the accepted op set at frontier F (post §9/§10 admission), an
evaluation policy P (`policy_id + version`). All rules are
set-predicates over judgments + causal ancestry — deterministic given
(ops, F, P). `authorized(j)` means j passed admission AND P counts its
actor class for its verdict (§11.4).

```
status(c, F, P):
 1. ∃ authorized retract|retire targeting c              → retired
 2. else ∃ authorized supersede(c → r) with
      status(r, F, P) == accepted and r ∉ cycle(c)       → superseded
    (supersession cycles: every claim in the cycle       → disputed)
 3. else ∃ authorized dispute d on c with no authorized
      owner-class accept a such that d ∈ ancestors(a)    → disputed
 4. else ∃ authorized accept                             → accepted
 5. else                                                 → candidate
```

- Concurrent Accept and Dispute both survive and both surface; rule 3
  makes the *derived* answer conservative (disputed) until an
  owner-class accept causally answers the dispute.
- Validity is orthogonal: a claim past `valid_until`/`expires_at`
  (compared against receipt-time, X1 discipline) is excluded from
  retrieval regardless of status.
- `supersedes[]` in a body NEVER changes status (advisory lineage).
- Declassify adjusts effective classification (§11.5), not status.

### 11.4 Policies (content-addressed, two built-ins)

A policy is a content-addressed document
`{ policy_id, version, rules: [(verdict, kinds, space_class, actor_classes)] }`.
v1 ships two embedded built-ins with pinned hashes:

- **`workflow-v1`**: self-accept counts for `observation`/`episode`
  authored by the same session's actor in designated workflow spaces;
  `decision` only with the session's own approval-trail evidence;
  owner-class judgments count for everything. `preference` and
  `procedure` acceptance: owner-class only (RFC §6.2).
- **`owner-v1`**: only owner/curator-class judgments count.

Status derivation always names its policy; projections stamp it
(RFC §13).

### 11.5 Classification and flows (RFC §6.2/§6.3)

`effective_class(c) = max(space minimum, writer claim,
max(evidence taint), curator raises) − authorized declassifications` —
monotone conservative, destination-aware. Retrieval into a remote
model's context is egress: the provider/model destination participates
in authorization like any flow endpoint.

`memory.export` (op type `m.export`) is the **stateless composite
export**: body `{ sources: [claim op_hashes], to { plane, zone,
space }, class_ceiling }`. Admission: source read authority on every
source ∧ a matching flow grant (or two-sided cross-plane pair, RFC
§6.3) ∧ `max(effective_class(sources)) ≤ min(flow ceiling, grant
ceiling)` ∧ destination minima. Cross-plane import lands as candidate;
source acceptance does not transfer; source classification travels as
a lower bound in provenance. Cross-call taint refusal is bounded
defense in depth; paraphrase through an unconstrained session remains
a stated residual (RFC §6.3).

### 11.6 Retrieval eligibility (service rule, pinned here)

Auto-context = `status == accepted` **∧** live pin (unexpired, no
Unpin) **∧** the pin's `accepted_under` judgment is still the winning
acceptance under the evaluation policy **∧** no unresolved dispute/
retract/supersede/expiry — else fail closed (RFC §6.1). `procedure`
and `preference` never ride ordinary retrieval; pins and curation
flows only (RFC §6.5). Candidates are excluded by default and marked
when opted into.

## 12. Migration invariant and versioning

- **M1.** Operations are immutable and never re-signed; containers,
  manifests, and projections may be rewritten; P1-to-P2 migration
  re-encapsulates the **exact stored operation bytes** (umbrella §15.1).
- **M2.** Version bumps: `v` fields version object *shapes*;
  `operation_version` versions op-type semantics; `protocol_version`
  in the header versions the envelope. Unknown effectful versions fail
  closed (E6).
- **M3.** The two built-in policies, the reserved control constants,
  the deny-reason enum, and the algorithm-ID vocabulary are frozen by
  golden vectors; additions are append-only.

## 13. Golden vectors and required tests

Vectors live in `crates/owner-plane-core/vectors/*.json` and run as
**required keyless CI** in both Rust and a browser (WebCrypto) harness
(RFC §13 — the crate and its lanes are added to CI by name; hand-listed
lanes do not auto-discover new crates).

**Vector families (freeze gate):**

1. **Encoding** — one per object family (§4), plus adversarial:
   unsorted keys, non-minimal int, indefinite length, unknown key,
   float — all MUST reject.
2. **Hash/domain** — `H_type` outputs per family; cross-tag
   verification MUST fail.
3. **Signatures** — Ed25519 sign+verify (deterministic); P-256 verify
   incl. a **high-S rejection** vector and a normalized-accept pair.
4. **HPKE** — wrap/unwrap with fixed ephemeral keys (RFC 9180
   test-vector style); wrong-info rejection.
5. **Item crypto** — encrypt/decrypt round-trip; wrong-AAD rejection;
   DEK-wrap binding rejection (wrong `ct_hash`, wrong epoch);
   tombstone shape.
6. **Frontier** — canonical bytes + hash for 0/1/n-writer maps.
7. **Control fold** — scripted chains: genesis bundle → enroll → grant
   → succession; competing AdminSuccessions (freeze) →
   RecoverySuccession resolution; stale-root rejection; hosted-genesis
   ceiling (enroll-native rejected, flow-grant rejected, effect-grant
   rejected) → trusted re-root lifts for new certs; DrillProof;
   replayed op idempotence; genesis-arm-after-seq-1 rejection.
8. **Expiry/budget** — receipt-qualified acceptance at/past deadline;
   missing-receipt quarantine; online-lease max-age/skew; budget
   exhaustion mid-chain (deterministic cut); cert expiry late arrival.
9. **Memory fold** — every §11.3 rule row; concurrent Accept+Dispute;
   owner-accept resolving a dispute causally; supersede chain + cycle;
   advisory `supersedes[]` non-effect; assert-as-claim+self-accept
   under `workflow-v1` vs rejected under `owner-v1`; pin/unpin +
   `accepted_under` invalidation; declassify precedence; expiry
   exclusion.
10. **IAM** — ≥12 DecisionInput vectors covering every deny reason at
    least once, `memory.search`-allowed with `memory.read`-denied,
    provenance-ceiling denial, flow composite allow/deny.

**Test map (RFC §16 lines this document discharges):** golden vectors
as required keyless CI; genesis bootstrap + first cert/grant via
GenesisAuthority; stale admin epoch; competing successor histories +
independent-recovery precedence; signing-key continuation fork (writer
generation, §6.2); per-item DEK erase incl. erase-vs-checkpoint race
and backup-restore (erase family + L-format crash tests); capability
late-arrival vs deadline; per-writer budget exhaustion; strict/lenient
zone policy; actor/signer mismatch; hosted ceiling incl. attempted
self-elevation; `assert` vs `propose` folds; pin revocation;
under/declassification; composite export incl. read-A+write-B
refusal; local log crash atomicity + browser transaction/multi-tab
(L1–L5). Remaining §16 lines belong to later gates.

## 14. Adversary boundaries (what each mechanism does NOT do)

| Mechanism | Defends against | Explicitly does not |
|---|---|---|
| Domain-separated signing + closed encodings | cross-type/cross-context confusion, malleability forks (S1) | a compromised signer signing bad content |
| AuthorizationProof + control reducer | bootstrap circularity, stale roots, replayed ceremonies | two holders of the same recovery secret (C3 freeze) |
| Independent recovery authority | later hosted/admin compromise self-succession | genesis-time hosted TCB compromise (admitted lane, RFC §3.5); loss of both secrets |
| Acceptance-deadline expiry | wall-clock forgery, HLC poisoning | distinguishing honest late writes from backdated ones (policy X5 chooses the window) |
| Item DEK + zone-rotation erase | erased-item recovery by cooperative replicas/backups | retained KEKs/ciphertext/plaintext; Connect withholding |
| Typed IAM ∩ | ambient authority, coarse-role leakage into the plane | paraphrase egress by an unconstrained session (RFC §6.3 residual) |
| Memory fold conservatism | poisoning via self-accepted truth, silent supersession | bad-but-authorized owner judgments (audit, not prevention) |
| Local log CRC + truncation recovery | torn writes | media corruption of committed history (that is what replicas are for) |

## 15. Decisions made in this document (ratify or amend)

Technical picks with rationale — flag any to change; silence at freeze
ratifies:

1. **Deterministic CBOR (E1–E6), text keys, SHA-256 everywhere** —
   §1/§2 rationale.
2. **Header-only signature + `body_hash` binding (O1)** — enables
   header-only validation and detached-body projections.
3. **Low-S ECDSA mandatory (S1)** — anti-fork-framing; golden-vector
   pinned.
4. **Random 96-bit item nonce** despite single-use DEKs — §5.1.
5. **Reserved control namespace over a second header (N1)** — one
   envelope shape, one signing domain.
6. **Single-writer control chain v1 (§7.1)**; control fork = freeze +
   recovery-only resolution (C2/C3).
7. **Recovery v1 = owner-held 24-word BIP39-derived offline Ed25519
   key, commitment-only in genesis (R1–R3)**; thresholds deferred to
   governance v2. ← *the one §App-C #3 sub-decision made here.*
8. **Phrase-loss-with-healthy-admin is not smoothed in v1** (§8.2) —
   accepting a re-root migration; smoothing it later is additive.
9. **Marker file `.intendant-space`** (N2).
10. **Classification ladder: public < internal < private < sensitive**
    (4 levels, policy-extensible).
11. **Two built-in judgment policies (`workflow-v1`, `owner-v1`),
    content-addressed** (§11.4).
12. **Expiry working position adopted as normative (X1–X5)** with the
    App-C #2 offline-usage confirmation as a pre-freeze test scenario,
    not a redesign.

**Still owner-open (unchanged from umbrella App C):** drill cadence
