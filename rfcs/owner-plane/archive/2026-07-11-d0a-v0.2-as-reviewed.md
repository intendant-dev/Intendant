## 0. Scope and non-goals

Unchanged from v0.1 in outline; this revision completes it into an
implementable protocol: exact bytes (CDDL, Appendix A), validation
tables, a bootable control lifecycle, writer lineages, capability-epoch
lifecycle, a real time/lease authority, the service-edge/portable-
admission split, executable Memory admission and folds, and portable
storage. Out of scope stays with the later gates: segment containers,
public manifests and leakage docs, recipient packaging at rest on
Connect, proof distribution, witness networks, multiwriter torture, the
Connect plane-store contract (D0-B); Agenda reducers (D0-Agenda-Data /
-Effects). D0-A owns types consumed later (frontier, receipts); later
gates own their transport.

## 1. Encoding profile

All signed or hashed objects use **RFC 8949 §4.2.1 Core Deterministic
Encoding** (owner ruling D-13), restricted further:

- **E1.** Integers shortest-form; **no floats, no bignums** in signed
  content.
- **E2.** Definite lengths only.
- **E3.** Map keys are UTF-8 text in `snake_case`, sorted by **bytewise
  lexicographic order of their canonical encodings** (the RFC 8949 §4.2.1
  rule — v0.1's "length-first" description was RFC 7049's superseded
  ordering and is void). Signed objects use **maps with text keys
  only**; anything keyed by binary identity is a **sorted,
  duplicate-free array** of records (see Frontier, §4.6).
- **E4.** No CBOR tags, no `null`, no `undefined`. Optional fields are
  omitted when absent. **There are no default values anywhere in this
  protocol** — every present field is meaningful, every required field
  is required; validators reject unknown keys in signed objects,
  out-of-order keys, non-shortest integers, duplicate keys.
- **E5.** Hashes/keys/IDs are byte strings; enums are text from closed
  vocabularies; timestamps are unsigned Unix milliseconds.
- **E6.** Versioning: every **top-level** signed/hashed object carries
  `v` (its shape version). Operation **bodies do not carry `v`** — the
  envelope's `operation_type` + `operation_version` version them (one
  version authority). Unknown `v`/`operation_version` on an effectful
  object fails closed.
- **E7 (sets).** Every array documented as a *set* is sorted by the
  bytewise order of its members' canonical encodings and duplicate-free;
  violations are parse failures.
- **E8 (hard caps — deterministic parsing, not operational policy).**

  | Cap | Value |
  |---|---|
  | canonical control object | ≤ 64 KiB |
  | canonical tenant operation (triple) | ≤ 256 KiB |
  | `statement` text | ≤ 16 KiB |
  | `causal_references` | ≤ 64 |
  | `evidence` refs | ≤ 32 |
  | export `sources` | ≤ 128 |
  | KEK wraps per op | ≤ 1024 |
  | nesting depth | ≤ 8 |
  | text field (other) | ≤ 4 KiB |

- **E9 (strict decoding).** Implementations MUST either use a strict
  raw decoder that enforces E1–E8 during parse, or parse
  duplicate-aware and prove `re-encode(parse(bytes)) == bytes`. An
  ordinary object deserializer alone is non-conformant.
- **E10 (parse/admission outcomes).** Every parse/validation failure
  maps to one member of the closed outcome enum (§10.4). No free-text
  rejection classes.

**Domain separation** (unchanged, one addition): 

```
msg(type, x) = "intendant/" || type || "/v1" || 0x00 || canonical(x)
H_type(x)    = SHA-256(msg(type, x))
```

Type tags: `genesis`, `cert`, `grant`, `op`, `body`, `frontier`,
`receipt`, `lease`, `drill`, `kek`, `item`, `policy`, `key`, `evrec`.
**Key IDs**: `key_id = H_key({alg, pk})` — the `key` tag is the hash
domain for all key identifiers.

## 2. Algorithm suite v1 (`suite-v1`)

Unchanged from v0.1 (SHA-256; Ed25519 + low-S P-256; HPKE base
DHKEM(P-256, HKDF-SHA256)/HKDF-SHA256/AES-256-GCM as `hpke-p256-v1`;
AES-256-GCM content AEAD `a256gcm`), with these completions:

- **S1 (low-S)** as v0.1 (anti-fork-framing; vector-pinned).
- **S5 (HPKE input validation).** Per RFC 9180: P-256 public-key inputs
  MUST be validated on-curve and non-identity; a DH output of the point
  at infinity is a failure. Malformed-input vectors are required (§13).
- **S6 (BIP39 recovery derivation, exact).** English wordlist, 24 words
  (256-bit entropy + 8-bit checksum). Mnemonic and passphrase are
  UTF-8 **NFKD-normalized**; the passphrase is the **empty string**.
  `seed = PBKDF2-HMAC-SHA512(password = mnemonic, salt = "mnemonic",
  iterations = 2048, dkLen = 64)`.
  `sk_ed25519_seed = HKDF-Expand(HKDF-Extract(salt =
  "intendant/recovery/v1", IKM = seed), info = "ed25519-seed", L = 32)`.
  Checksum-invalid mnemonics are rejected before derivation.
- **S7 (key custody contract).** Per class:

  | Class | Signing/KEM key custody |
  |---|---|
  | hosted-browser / owner-browser | non-extractable WebCrypto keys in IndexedDB (platform-sealed) |
  | native-app | OS keystore (Keychain / DPAPI / keyctl) |
  | daemon | passkey-sealed vault custody where configured; else OS keystore; file fallback is mode-0600 **and marked degraded in the certificate evidence** |
  | recovery | paper/offline only; never persisted by software |

  **Durable plane keys and plane plaintext NEVER enter
  `intendant-runtime`** — the runtime/controller boundary is a protocol
  invariant here, not just repo doctrine.

## 3. Identifiers

As v0.1 with these changes:

- **N1 (reserved IDs, exact).** Real 16-byte IDs (ULIDs and random IDs)
  MUST NOT have their first 8 bytes all zero; validators reject such
  IDs from writers. The reserved control constants are:

  ```
  CTRL_ZONE    = h'00000000 00000000 00000000 00000001'
  CTRL_SPACE   = h'00000000 00000000 00000000 00000002'
  CTRL_LINEAGE = h'00000000 00000000 00000000 00000003'
  ```

- **N2** (`.intendant-space` marker) unchanged.
- **N3 (writer identity is a lineage, not a local mint).** The header's
  writer field is `writer: { lineage: bytes(16), gen: uint }`.
  Lineages are **control-authorized objects** (§7); generations are
  successors within a lineage (§9.3). v0.1's locally minted `writer_id`
  is void — it allowed budget/cutoff evasion by re-minting.

## 4. Core objects

Normative CDDL for everything in this section: **Appendix A**. Prose
here states semantics and deltas from v0.1.

### 4.1 GenesisDescriptor (`genesis`)

As v0.1 (root sig key; `recovery_commitment = H_drill(recovery pk)`;
`governance {v:1, kind:"single-owner"}`; `provenance`; `suite`),
unchanged. `plane_id = H_genesis(descriptor)`.

### 4.2 DeviceCertificate (`cert`)

As v0.1, with:

- `device_id` is **stable across certificate renewals** and is what
  grants bind (§4.3) — renewal never orphans grants.
- Renewal: an `EnrollDevice` control op carrying the same `device_id`
  and `renews: old_cert_hash`. On acceptance the old certificate is
  **superseded**: valid for validating operations whose acceptance
  receipts predate the renewal; invalid for newly accepted operations.
  Old-cert revocation for cause remains `RevokeDevice`.
- `evidence_hash` commits a per-class evidence document; **v1 evidence
  documents are opaque bytes** with a declared `media_type` (owner may
  attach anything; validators check the hash only). Structured evidence
  schemas are a later, additive upgrade.

### 4.3 CapabilityGrant (`grant`)

Delta from v0.1:

- `subject_device: bytes(16)` (**device_id**, not cert hash) —
  validation resolves the signer's current, non-superseded,
  non-revoked certificate for that device.
- `lineage: bytes(16)` — **required on every write-capable grant**; the
  grant binds the writer lineage whose budgets/cutoffs it feeds.
- `max_generations: uint` (default absent ⇒ 8) — self-service
  generation advances per capability epoch before a control
  re-authorization is required (§9.3).
- `max_age_ms: uint` — required when `online_lease = true` (v0.1
  referenced it without defining it).
- `budget {max_ops, max_bytes}`: accounted per
  `(grant_id, lineage, capability_epoch)`; `max_bytes` counts the
  **canonical encoded bytes of the full SignedOperation triple**; each
  operation cites exactly one grant, so accounting is unambiguous.
  A capability-epoch bump (§9.4) deliberately resets budget accounting
  (it is an admin act).
- Class ceilings are constrained by the **device-class ceiling table**
  (§7.6) on *every* plane — the hosted-plane ceiling (§7.5) is a
  second, independent cap.

### 4.4 AuthorizationProof

As v0.1 (arms `dev`, `genesis`, `admin`, `recovery`), unchanged in
shape. Arm admissibility: §7.2.

### 4.5 SignedOperation envelope (`op`, `body`)

Delta from v0.1:

```
header {
  v: 1,                                # THE protocol version (M2 refers here)
  tenant: "memory" | "agenda" | "ctrl",
  plane_id, zone_id, space_id,
  authored_kek_epoch: uint,            # the only KEK/crypto epoch in signed
                                       # bytes (capability/admin epochs are
                                       # also signed — v0.1's "only epoch"
                                       # wording was wrong)
  capability_epoch: uint,              # zone capability epoch, §9.4
  signer_alg, signer_key_id,           # key_id = H_key({alg,pk}), §1
  writer: { lineage: bytes(16), gen: uint },
  actor { … as v0.1 … },
  authorization_proof, request_id,
  writer_sequence: uint, previous_writer_hash: bytes(32),
  causal_references: [* bytes(32)],    # set, E7
  created_hlc: [uint, uint],
  operation_type: text, operation_version: uint,
  body_hash: bytes(32) }
```

- **O1/O2** (header-only signature; `op_hash` over the triple)
  unchanged.
- **O3 (restated).** `authored_kek_epoch` is the only **KEK** epoch in
  signed bytes; the mutable `key_wrap_epoch` lives in the item wrap
  (§5.3).
- **O6 (chain scope).** Per-writer chains are per
  `(plane, zone, lineage, gen)`. A generation's first operation MUST be
  the `w.gen` generation-advance record (§9.3);
  `previous_writer_hash` of that first op is the sentinel
  `SHA-256("intendant/genstart/v1" || 0x00 || lineage || gen_be64)`.
- **O7 (control ops).** Control operations use the N1 constants for
  zone/space/lineage and form the single control chain (§7.1).

### 4.6 Frontier (`frontier`)

E3-conformant restatement (v0.1's byte-keyed map violated its own
profile):

```
Frontier = { v: 1, heads: [* Head] }        # set per E7
Head     = { lineage: bytes(16), gen: uint, seq: uint, op: bytes(32) }
```

Sorted by `(lineage, gen)` encoded bytes; at most one head per
`(lineage, gen)`. The canonical frontier type freezes here; D0-B adds
Merkle compression and witness proofs over it.

### 4.7 Receipts and lease proofs (`receipt`, `lease`)

Reworked (review blocker 4 — v0.1's "plane devices always qualify" let
an operation's own signer backdate its own deadline receipt):

```
StorageReceipt    { v:1, kind:"storage",  issuer_key_id, subject: item_addr,
                    size: uint, seen_ms: uint, issuer_seq: uint }
AcceptanceReceipt { v:1, kind:"accept",   issuer_key_id, subject: item_addr,
                    seen_ms: uint, issuer_seq: uint }
ReplicaAck        { v:1, kind:"replica",  issuer_key_id,
                    frontier_hash: bytes(32), seen_ms, issuer_seq }
CheckpointWitness { v:1, kind:"witness",  issuer_key_id,
                    checkpoint: bytes(32), seen_ms, issuer_seq }
LeaseProof        { v:1, kind:"lease",    issuer_key_id, grant_id,
                    lineage, issued_ms, expires_ms,
                    ctrl_frontier_hash: bytes(32), issuer_seq }
```

- **T1 (subjects are item addresses).** Receipts name `item_addr`
  (§5.3), never zone-private `op_hash` — Connect and other
  non-members learn only ciphertext identities; zone members bind
  item → operation locally (§5.6).
- **T2 (witness qualification, owner ruling D-11).** Qualified time
  witnesses for an operation are, per zone policy: the plane's other
  enrolled devices **excluding the operation's signer** (a signer
  NEVER qualifies for its own deadline evidence), plus Connect iff
  `accept_connect_time = true`. `ZonePolicy.time_witnesses` may narrow
  the set to named `key_id`s or classes.
- **T3 (anti-replay).** `issuer_seq` is strictly monotone per issuer
  key; duplicate `(issuer_key_id, issuer_seq)` with different bytes is
  issuer-fork evidence; receipts are idempotent by exact bytes.
- **T4 (issuer rotation).** Receipts validate against the issuer's
  certificate state at `seen_ms` under the same acceptance-deadline
  discipline as everything else.
- **T5 (lease objects exist now).** `online_lease` grants require a
  `LeaseProof` from a qualified witness, `expires_ms − issued_ms ≤
  grant.max_age_ms`, and `ctrl_frontier_hash` naming a control frontier
  the validator has. Validator-local time may gate a **live request**;
  it never changes the replicated fold (§9.2).

## 5. Item crypto

### 5.1 DEK and AEAD (as v0.1)

Single-use 32-byte DEK per item; AES-256-GCM; AAD
`"intendant/item/v1" || 0x00 || plane_id || zone_id`. Plaintext is the
exact SignedOperation triple bytes.

### 5.2 Zone KEKs (as v0.1)

Per (zone, kek_epoch); distribution via `hpke-p256-v1` with the §5.2
info string; every wrap carries its KEM algorithm ID.

### 5.3 Item records — immutable core, mutable wrap

Split (review blocker 1 — the content address must cover the complete
immutable ciphertext record, and `op_hash` must not ride the shippable
outer record):

```
ItemCore  { v:1, aead:"a256gcm", nonce: bytes(12), ct: bytes }   # immutable
item_addr = H_item(ItemCore)                                     # the content address

ItemWrap  { v:1, item_addr, key_wrap_epoch: uint,
            wrap_nonce: bytes(12), wrapped_dek: bytes }           # mutable
```

- **I1.** `item_addr` (over the complete core: version, AEAD ID, nonce,
  ciphertext) is the only item identity that leaves the zone. `op_hash`
  lives in the **local private index** (§5.6) — never in ItemCore or
  ItemWrap. If D0-B ships these records verbatim, Connect learns
  ciphertext identities only.
- **I2 (wrapper nonces, owner ruling D-10 — deterministic).**
  `wrap_nonce = HKDF-Expand(HKDF-Extract(salt =
  "intendant/wrapnonce/v1", IKM = KEK), info = item_addr, L = 12)`.
  Under one KEK, nonces collide only if `item_addr`s collide; rewrap of
  the same item under the same epoch is byte-idempotent. No invocation
  bookkeeping.
- **I3 (DEK-wrap AAD)** as v0.1, with `ct_hash` replaced by
  `item_addr`: `"intendant/dekwrap/v1" || 0x00 || plane_id || zone_id
  || kek_epoch_be64 || item_addr`.
- **I4 (current wrapper).** The current wrapper for an item is the
  ItemWrap whose `key_wrap_epoch` equals the zone's **accepted current
  KEK epoch** in control state; wrappers for fenced epochs are not
  served and are GC targets (§5.5).
- **I5 (validation after decryption).** A consumer MUST recompute
  `body_hash`, verify the header signature, recompute `op_hash`, and
  check the local index binding before treating content as an accepted
  operation.

### 5.4 Erase workflow (owner ruling D-5 — request + batched rotation)

Two lanes:

1. **Erase request** — a tenant Memory operation (`m.erase_request`,
   §11.1) naming claim op_hashes. On acceptance the items are
   **immediately excluded from retrieval** (a retrieval-expiry effect);
   requests accumulate.
2. **Batched rotation ceremony** — one admin `KekRotation` control op
   (§7.1) carrying `erase_manifest: [item_addr]` (the accumulated
   requests) plus wrap-adds for members. One ceremony erases any number
   of items; **no per-item admin ceremonies** (v0.1's `EraseAuthorize`
   is void).

### 5.5 Crash-safe local erase state machine

Persisted states; recovery re-enters at the last durable state:

```
1 RotationAccepted   (control op durable: new epoch, erase_manifest)
2 Fenced             (zone-log FENCE record: old epoch no longer served)
3 RewrapProgress     (idempotent survivor rewraps; deterministic nonces
                      make replays byte-identical)
4 RewrapComplete     (zone-log checkpoint record covering the survivor set)
5 OldKekDestroyed    (local key store deletion — ONLY after 4 is durable)
6 Tombstoned         (EraseTombstone records for manifest items)
```

Crash between 2–5 never resurrects: fenced wrappers are not served
(I4); old-KEK destruction strictly follows a durable RewrapComplete.
D0-B adds replica acknowledgements and distributed GC on top;
local-crash resurrection is impossible from D0-A alone. Erase tests
MUST cover index, view, checkpoint, and backup copies (§13).

### 5.6 Local private index

`op_hash ↔ item_addr` (+ per-claim retrieval flags) is a **local,
rebuildable index**: on the daemon it lives inside the plane store
(box-trust; sequencing metadata may be plaintext, operation *content*
may not — §6.1); in the browser, an IndexedDB store. It is rebuilt by
decrypt-and-scan; it is removed/rebuilt on revocation and erasure with
every other projection (RFC §6.2).

## 6. Local durable storage

### 6.1 Daemon logs — tenant logs are ciphertext-only

**The v0.1 format was defective: it persisted plaintext
`SignedOperation` records beside encrypted items, defeating DEK
erasure.** v0.2:

- **Tenant zone log** record types (no plaintext-operation type
  exists):

  ```
  0x11 ItemCommit   { core: ItemCore, wrap: ItemWrap,
                      lineage, gen, seq }      # sequencing metadata is
                                               # plaintext-local (box trust);
                                               # operation CONTENT never is
  0x12 ItemRewrap   { new wrap }
  0x13 Fence        { kek_epoch }
  0x14 RewrapDone   { kek_epoch, covers: frontier_hash }
  0x15 Tombstone    { EraseTombstone }
  0x16 Receipt      { any §4.7 object }
  0x17 OutboxMark   { through: {lineage, gen, seq} }
  ```

- **Control log** (separate file, distinct schema): plaintext control
  operations, plane-scoped — they are the validation material every
  device needs. Type `0x01 CtrlOp { SignedOperation bytes }` is legal
  **only** here; a tenant log containing it is corrupt.
- **One logical durable commit**: sequence allocation + ItemCommit +
  outbox state advance are a single appended record (0x11 carries all
  three); there is no window where a sequence is allocated but its
  encrypted item is not durable.

### 6.2 File format and recovery

```
file  = magic "IPLOG1" || format_version u8 || kind u8 (tenant|ctrl)
        || plane_id || zone_id || records…
record = len u32-le || crc32c u32-le (over type‖payload)
        || type u8 || payload
```

- **L1 (commit + platform durability).** A record is committed when
  durably flushed via the **platform durability primitive**: Unix
  `fsync` (file, and directory on creation); Windows
  `FlushFileBuffers`; atomic replace = `rename` / `MoveFileEx`
  (`MOVEFILE_REPLACE_EXISTING | MOVEFILE_WRITE_THROUGH`). "fsync" in
  this document means this abstraction.
- **L2 (torn tail vs middle corruption).** Recovery scans forward. An
  invalid record **at the tail** (short read or bad CRC with nothing
  valid after it) is truncated. An invalid record **followed by valid
  records** is media corruption: the log is quarantined read-only and
  the store enters replica-rebuild; it is never silently repaired.
- **L3 (cross-process exclusion).** One exclusive advisory lock per
  plane store (Unix `flock`, Windows `LockFileEx`) — this machine runs
  multiple controller processes as a matter of course; a process that
  fails to acquire the lock gets read-only access, never a second
  appender.

### 6.3 Browser mapping

- Object stores `items`, `wraps`, `receipts`, `frontier`, `index`,
  `meta` — **no plaintext tenant-operation store** (v0.1's `ops` store
  is void); the browser holds zone keys and decrypts on demand, caching
  plaintext only in memory.
- One IndexedDB transaction per logical commit. IndexedDB atomicity is
  real; **strict durability is a hint, not a power-loss guarantee**:
  the client requests `navigator.storage.persist()` and
  `durability: "strict"` where supported, and the degraded floor is the
  umbrella's zero-daemon floor (RFC §4.4) — stated, not papered over.
- Web Locks (`navigator.locks`) provide the tab writer lease —
  **cooperative exclusion within a storage bucket, not authorization**;
  feature-detect, and without Web Locks the tab is read-only.
- Writer generations per §9.3 (lineage successors, not new budget
  principals).

## 7. Control plane

### 7.1 Operations (single control chain, as v0.1 §7.1)

Revised operation set — compound lifecycle ops restore bootability
without widening the ceiling (review blocker 2; owner rulings D-3,
D-4):

```
GenesisBundle        # seq 1, atomic: descriptor
                     # + first DeviceCertificate (+ custody evidence)
                     # + first WriterLineage
                     # + ZoneCreate("private", epoch-1 KEK, wrap to first device)
                     # + SpaceCreate(home) + SpaceCreate(audit)
                     # + ZonePolicyInstall(defaults)
                     # + first CapabilityGrant (class-ceiling-constrained)
EnrollDevice         # compound: cert (new or renews:old_cert_hash)
                     # + grants[] + WriterLineage create-or-successor
                     # + WrapAdd per granted zone (current epoch)
RevokeDevice         # compound: cert revocation + grant revocations
                     # + lineage cutoff + KekRotation(exclusion) per
                     #   zone the device could decrypt
WrapAdd              # current-epoch KEK wrap for an enrolled device
                     # (D-4: O(1) joins; rotation reserved for exclusion)
ZoneCreate           # zone + epoch-1 KEK + initial wraps (never wrapless)
SpaceCreate          # space_id, zone, name_hash, class_minimum,
                     # status_policy (content-addressed policy hash)
SpacePolicySet       # rebind class_minimum / status_policy
SpaceRetire
ZonePolicyInstall    # full ZonePolicy object (Appendix A), versioned
IssueGrant / RevokeGrant
CapabilityEpochBump  # zone_id, new_epoch (§9.4)
LineageReauthorize   # lineage, new max_generations window (§9.3)
KekRotation          # zone, new_epoch, wrap_adds[], erase_manifest[]
CutoffSet            # lineage, accepted_through {gen, seq, op}
AbandonWriter        # lineage, at {gen, seq, op}
CheckpointCommit     # checkpoint_hash, covers: frontier_hash, gc_fence
AdminSuccession      # new_admin {alg,pk}, epoch (explicit integer)
RecoverySuccession   # §7.4
DrillProof           # §8.3
```

`HostedEnroll`/`HostedRevoke` are the same `EnrollDevice`/`RevokeDevice`
ops constrained by the §7.5 ceiling — not separate types.

### 7.2 Authorization matrix

As v0.1 (arms × ops) extended to the new ops: everything except
`GenesisBundle` (genesis arm, seq 1 only), `RecoverySuccession` and
`DrillProof` (recovery arm only) requires the current `admin` arm; the
`dev` arm never authorizes control ops; `ctrl` is never grantable.

### 7.3 Bootability invariant

> After `GenesisBundle` alone, the first device MUST be able to: write
> a Memory claim into the `private` zone's `home` space; read it back;
> enroll a second device (`EnrollDevice` with `WrapAdd`) that can then
> read the same history; and revoke that device (`RevokeDevice`) such
> that it cannot decrypt anything written after the rotation.

This walkthrough is a required vector (§13, family 7) for both
`trusted` and `hosted` genesis.

### 7.4 Epochs, precedence, forks, recovery — exact rules

C1/C2/C4/C5 as v0.1 (consecutive admin epochs; duplicate control
sequence = freeze; stale roots; replay). C3 is replaced with a
deterministic rule (v0.1's `max(observed)+1` varied by replica):

- **C3′ (recovery resolution).** `RecoverySuccession` body:

  ```
  { base: { seq: uint, op: bytes(32) },     # last control head the
                                            # resolver keeps
    epoch: uint, repoch: uint,              # explicit integers
    new_admin: { alg, pk },
    new_recovery_commitment: bytes(32),
    tenant_cutoffs: [* { lineage, accepted_through {gen, seq, op} }] }
  ```

  Validity: signed by the revealed recovery key matching the current
  commitment; `repoch = current_repoch + 1`; `epoch` strictly greater
  than the epoch at `base`. **Rule: after acceptance, the control chain
  is exactly `ancestors(base) ∪ {this op} ∪ descendants(this op)`;
  every control operation outside that set is retired** — enumeration
  of abandoned heads is unnecessary and unknown branches are cut
  automatically. Deterministic from the operation's own bytes.
- **Branch-cut default (owner ruling D-8, strict).** Tenant operations
  from cut branches' lineages past `base` are quarantined for
  re-proposal per `tenant_cutoffs`; a lenient `ZonePolicy` may widen
  acceptance, never narrow the quarantine below the cutoffs.
- **Honest control forks (owner ruling D-7).** Any control fork —
  including one produced by an honest crash/restore — requires the
  recovery ceremony. Ratified: control is the authority plane; honest
  and hostile forks are indistinguishable; L1 durability makes honest
  ones vanishingly rare.
- Competing `RecoverySuccession`s at the same `repoch` remain the
  declared plane-compromise freeze (v0.1 C3 tail).

### 7.5 Hosted-plane ceiling (revised per owner ruling D-2/D-3)

While `genesis.provenance == hosted` and no trusted-client
`RecoverySuccession` has been accepted:

- **(a)** only `hosted-browser`-class certificates may be enrolled;
- **(b)** grants MAY carry `class_ceiling` up to **`sensitive`** for
  `search/read/propose/assert/write`-class verbs in the plane's own
  zones — **a hosted diary stores private content with accurate
  labels** (v0.1's `internal` cap contradicted the product and is
  void). What stays excluded: `memory.export` flows, declassification,
  instruction-grade curation (`procedure`/`preference` acceptance,
  graduation), every Agenda effect verb, and `memory.admin`;
- **(c)** admissible control operations: `EnrollDevice`
  (hosted-browser class, compound — the second browser gets its grant,
  lineage, and wrap in the same op), `RevokeDevice` (compound — the
  revoked browser is excluded by its rotation), `WrapAdd` between
  enrolled hosted browsers, and `DrillProof`. **No destructive
  maintenance**: no `CheckpointCommit` GC fences, no `KekRotation`
  with an `erase_manifest`, no `AbandonWriter` (D-3). Consequence,
  stated honestly: hosted-only planes get retrieval-exclusion erasure
  only; **cryptographic erasure arrives with the first trusted client
  or daemon** — key destruction performed by the admitted hosted TCB
  would be theater anyway.

### 7.6 Device-class ceiling table (every plane)

Independent of §7.5 — a hosted browser enrolled on a trusted plane is
still a hosted browser:

| Class | Max grant `class_ceiling` | Excluded verbs (never grantable) |

| owner-browser (owner-served dashboard) | sensitive | none (trusted client per RFC §3.5) |
| native-app | sensitive | none |
| daemon | sensitive | none |

## 9. Deterministic expiry, budgets, lineages, epochs

### 9.1 Acceptance deadlines (X1 revised)

An operation whose cited grant/cert carries `expiry_deadline_ms = D` is
accepted at fold time iff a **qualified** `AcceptanceReceipt` (T2 —
never the op's own signer) or a covering `CheckpointWitness` exists
with `seen_ms ≤ D`. Absent that: quarantine for re-proposal.
**No-witness environments (owner ruling D-11):** a zone policy either
declares `deadline_fallback = "fail-closed"` (default) or
`= "budgets"` — grants then rely on sequence/byte budgets (§4.3), which
need no time at all.

### 9.2 Local time

Validator-local clocks gate **live requests only** (online-lease
checks, T5, with `skew_ms = 300000`); the replicated fold never reads
a local clock. Semantic claim expiry uses the projection's explicit
`as_of_ms` (§11.6).

### 9.3 Writer lineages and generations (review blocker 3)

- Lineages are created/re-authorized only by control ops (§7.1) and
  bind `device_id`.
- A generation advance is **self-service within a bounded window**: the
  first operation of generation `g+1` MUST be `w.gen`
  (`operation_type = "w.gen"`, body
  `{ last_known: { gen, seq, op } | "unknown" }`), signed by the
  device's current cert. At most `grant.max_generations` advances per
  capability epoch; beyond that, admission denies (`lineage-gen`) until
  a `LineageReauthorize` control op. Honest browser eviction recovers
  without a ceremony; lineage-mint abuse cannot reset anything because:
- **Budgets, cutoffs, and quarantines key on the lineage**, summed
  across generations. A fresh generation inherits its lineage's
  consumed budget and its cutoffs. (The v0.1 evasion — exhaust, re-mint,
  reset — is closed by construction; a required vector proves it.)
- Signing-key continuation: cert renewal keeps `device_id`; the lineage
  binding is untouched; the first op under the new cert references the
  old head normally.

### 9.4 Capability epochs (review blocker 3, second half)

- **Scope**: per zone. `CapabilityEpochBump { zone_id, new_epoch }`
  (admin arm) advances it; consecutive integers.
- **Semantics**: an operation's `header.capability_epoch` MUST equal
  the zone's current epoch at authorship. On acceptance, ops citing an
  older epoch are quarantined (strict policy) or accepted within the
  policy's lenient window (`ZonePolicy.lenient_epochs: uint`, default
  0) — the umbrella's strict-vs-lenient knob, root-signed, never local
  UX.
- **Grants** record `capability_epoch` at issuance; a grant is usable
  while `zone_epoch − grant.capability_epoch ≤
  ZonePolicy.grant_epoch_slack` (default: unbounded — bumps are for
  soft revocation when wanted, not a mandatory treadmill).
- Budget accounting resets per epoch (§4.3) — deliberate, admin-gated.

## 10. Plane IAM — service edge vs portable admission (review blocker 5)

v0.1's single five-way intersection denied the flagship lanes (a
zero-daemon browser has no daemon session grant or MCP token; a native
controller-bound session has no bearer). v0.2 splits two layers:

### 10.1 Service-edge access shapes

| # | Shape | Local components intersected (beyond plane capability + request + class/provenance ceilings) |
|---|---|---|
| 1 | Direct enrolled device (browser/native/daemon; human owner) | — none (the device cert is the identity) |
| 2 | Controller-attested native supervised session | daemon-local session grant (controller-enforced; controller signs with `actor.attested_by`) |
| 3 | External process-tree bearer | live session token: unrevoked, session live, op ∈ token scope |
| 4 | Mediated peer / low-trust session | peer/session grant + token, per RFC §6.4 |

Each shape intersects **only its applicable components**; a component
that cannot exist for a shape is not "vacuously satisfied" — it is not
part of that shape's rule. Service read/search **quotas** are edge
policy (per shape/principal), separate from writer-sequence budgets —
a read has no replicated sequence.

### 10.2 Portable fold admission (replica-independent)

A separate algorithm using **only portable inputs** — certificates,
grants, control state, signed actor attestation, receipts, body/causal
invariants:

```
admit(op, control_state, receipts):
  parse strict (E1–E10)                       → parse outcomes
  cert: resolve signer cert for device; valid, not superseded-for-new,
        not revoked; class ceilings (§7.6)    → cert outcomes
  proof: arm admissible (§7.2); dev-arm grant scope
        (tenant∧zone∧space∧op∧kind); lineage binding; class ≤ ceilings;
        hosted-plane ceiling (§7.5)           → authz outcomes
  chain: writer chain intact; w.gen first-op rule; generation window;
        cutoffs; capability epoch (§9.4); budgets (lineage-summed)
                                              → chain outcomes
  time:  acceptance deadlines via receipts (§9.1); lease proofs (T5)
                                              → time outcomes
  body:  body_hash; operation_type/version known; body invariants
        (§11); causal references present      → body outcomes
```

No ephemeral state appears here — a remote replica can never check
another daemon's live token, so tokens and session grants are
edge-only by construction. **The reducer runs admission, not the
edge rule.**

### 10.3 DecisionInput (edge) — v0.1 §10.1 amended

`context.policy` is removed from caller control: the evaluation policy
is resolved from the **space's control-governed binding**
(`SpaceCreate.status_policy`), never from the request. `request.
classification` is likewise service-derived (§11.5), never
caller-trusted.

### 10.4 Closed outcome enum

```
parse:  malformed, oversized, depth, unknown-version, non-canonical
cert:   no-cert, cert-revoked, cert-superseded, cert-expired, class-excluded
authz:  proof-arm, no-grant, scope-tenant, scope-zone, scope-space,
        scope-op, scope-kind, class-ceiling, provenance-ceiling,
        hosted-ceiling, no-flow
chain:  fork, gen-first-op, lineage-gen, cutoff, capability-epoch, budget
time:   deadline-unreceipted, lease-missing, lease-stale
body:   body-hash, op-unknown, body-invariant, causal-missing, policy-missing
edge:   no-session, session-ended, no-token, token-scope, token-revoked, quota
```

Vector rule (§13): **at least one vector per enum member** plus at
least six allow-path combinations across shapes (v0.1's "≥12 covering
all reasons" was arithmetically incoherent and is void).

### 11.1 Operation admission table

| op_type | Edge permission | Body/actor invariants | Charge |
|---|---|---|---|
| `m.claim` | `memory.propose` | kind ∈ grant.kinds; sensitivity ≤ ceilings; evidence refs valid (§11.7) | 1 op, triple bytes |
| `m.claim` via **assert** | `memory.assert` | as above **and** kind ∈ {observation, episode}; space is workflow-class; actor == authoring session | as compound |

| `m.judge` accept (curatorial) | `memory.curate` | owner-class actor (§11.4) | 1 op |
| `m.judge` dispute | `memory.propose` (record) / status effect **owner-class only** | §11.3 note | 1 op |
| `m.judge` retract | author of target, or owner-class | v1 has **no un-retract** | 1 op |
| `m.judge` retire | owner-class `memory.curate` | | 1 op |
| `m.judge` supersede | owner-class; or author-within-workflow for own claims | replacement ref required | 1 op |
| `m.judge` raise_class | any authorized reader-writer of the claim's space | raises only | 1 op |
| `m.judge` declassify | owner-class `memory.curate` + grant flag `can_declassify` | §11.5 causal rule | 1 op |
| `m.pin` / `m.unpin` | owner-class `memory.curate` | unpin targets a pin op | 1 op |

- **`decision` claims are propose-only in v1** — no self-accept path
  exists until an approval-trail verifier does (adopted from review;
  the umbrella's allowance becomes reachable then, not before).
- **`m.assert` compound semantics**: one request → claim + self-accept
  judgment in **one durable commit** (§6.1); the judgment's
  `request_id = SHA-256("intendant/assert/v1" || claim.request_id)[0..16]`;
  idempotent re-submission completes the missing half if a crash split
  a non-conforming implementation.

### 11.2 Judgment bodies

As v0.1 (verdict variants incl. `supersede{target, replacement}`,
`declassify{target, new_class}`; every judgment carries
`policy {id, version}` provenance) plus the new `raise_class
{target, new_class}` variant.

### 11.3 Derived status fold

The v0.1 five-row precedence stands, with the counting rule made
explicit (review blocker 6):

- `authorized(j)` = j passed §10.2 admission **and** the evaluation
  policy P counts `(j.verdict, target.kind, space_class,
  actor_class(j))`.
- **Dispute counting (anti-suppression):** in both built-in policies,
  `dispute` counts toward status **only from owner-class actors**. A
  session-class dispute is recorded, surfaced in views and retrieval
  metadata, but does not flip status — a low-trust proposer cannot
  suppress accepted pinned context. (Required vector: unauthorized
  dispute does not change status.)
- **Revival (owner ruling D-9):** rule 2 requires the replacement to be
  currently `accepted`; if it later loses that status, the superseded
  claim revives automatically. Views surface revivals.
- Retract vs Retire both derive `retired` but are **distinct verdicts,
  surfaced distinctly** (author withdrawal vs curatorial retirement);
  v1 defines no un-retract.
- Missing/unknown policy hash on a space: the fold returns `candidate`
  for everything in that space (fail closed) with outcome
  `policy-missing`.

### 11.4 Policies and actor classes

- Policy objects: content-addressed
  (`H_policy`), full rule tables `(verdict × kinds × space_classes ×
  actor_classes)`. The two built-ins ship as exact CBOR constants with
  pinned hashes (Appendix A.9): `workflow-v1` (self-accept for
  observation/episode in workflow spaces by the authoring session;
  owner-class everything; owner-only dispute counting; owner-only
  procedure/preference acceptance) and `owner-v1` (owner-class only).
- **Space → policy binding is control state** (`SpaceCreate` /
  `SpacePolicySet`); callers never choose the policy (§10.3).
- Actor classes, derived — never asserted: `owner` (human actor on a
  no-exclusions device class §7.6 with `memory.curate`), `session`
  (controller-attested native session), `external` (process-tree
  bearer), `peer` (mediated). The derivation table is normative;
  `curator == owner` in v1.

### 11.5 Classification fold (deterministic)

```
base(c)     = max( space_minimum(space, control state),
                   c.sensitivity,
                   max(evidence floors; unresolvable ref ⇒ "sensitive") )
raised(c)   = max( base(c), all authorized raise_class verdicts )
effective(c)= if ∃ authorized declassify d whose causal ancestors
                 include EVERY authorized raise_class on c
              then max( base(c) capped by d.new_class, d.new_class )
              else raised(c)
```

A concurrent (non-descending) raise defeats a declassify —
conservative by construction. Requests never carry trusted
classification; the service derives it. Export ceilings compare
against `effective` (§11.8).

### 11.6 Views, expiry, and auto-context

- Projections evaluate at explicit `as_of_ms` (stamped, with data +
  control frontiers, reducer + policy versions, key epoch — RFC §13).
  Semantic expiry (`valid_until`, `expires_at`) is an `as_of_ms`
  comparison; acceptance receipts govern only whether a late operation
  *enters* the log (§9.1) — a claim accepted before its `expires_at`
  still expires from views on time.
- Auto-context: the v0.1 four-condition rule, clarified — the pin's
  `accepted_under` judgment must still **count toward** the accepted
  status under the space's bound policy (not necessarily be the unique
  winner). **A pin selects; it never grants: auto-context injection
  performs full per-item `memory.read` authorization for the receiving
  session** — an unauthorized session sees neither body nor existence.
- The owner-audit domain (RFC §6.2) is, in v1, the `audit` space in the
  `private` zone created at genesis; sensitive-space read audits are
  system-actor operations written there. A dedicated audit crypto
  domain is an explicit D0-B option, deferred.

### 11.7 Evidence references (typed)

```
EvidenceRef =
  { ns: "plane", op: bytes(32), zone: bytes(16), span?: text }
| { ns: "external", scheme: "session-log" | "url" | "file",
    locator_hash: bytes(32), digest: bytes(32) }
— both with: class_floor
```

Taint depth is **1** (a claim's taint reads its direct refs' floors
only — no transitive walks, no cycles); a dangling plane ref
contributes its recorded `class_floor`; an unresolvable external ref
contributes `sensitive`. `memory.evidence.read` authorizes
dereference; refs confer no read authority (RFC §6.3).

### 11.8 Export/import pair (two authorities, exact)

v0.1's single `m.export` could not express two planes' authority. v0.2:

```
m.export.release (source):
  { export_id: bytes(16), sources: [* bytes(32)],       # set, ≤128
    content_digest: bytes(32),        # H_body of the canonical released bundle
    to { plane_id, zone_id?, space_id? } | egress { kind, provider_id },
    class_floor,                      # = max effective(sources)
    expiry_deadline_ms }
m.import.claim (destination):
  { export_id, from_plane: bytes(32), content_digest,
    class_floor,                      # copied — the immutable lower bound
    …claim body fields… }
```

- Release admission: read authority on every source ∧ a flow grant
  matching the typed endpoint ∧ `class_floor ≤ min(flow ceiling, grant
  ceiling)` ∧ destination minima known to policy.
- Import admission: destination import-flagged propose grant; the
  `export_id + content_digest` pair must match a release the
  destination has verified; the new claim is a **candidate** with
  `provenance.import = {from_plane, export_id, digest}` and
  classification ≥ `class_floor` (lower bound; raise free, lower only
  via §11.5 declassification).
- Same-plane cross-zone uses the same pair (an integrated daemon may
  execute both in one local commit — still two operations in the
  logs), and the imported content is **re-encrypted as a new item
  under the destination zone's keys**.
- Typed flow endpoints include model/embedding/reflection egress:
  `{ egress: { kind: "model-provider" | "embedding" | "reflection",
  provider_id: text } }` — retrieval into a remote model's context is
  this flow, not a side door.
- Cross-call taint refusal remains bounded defense in depth; the
  paraphrase residual stays admitted (RFC §6.3) — no vector may claim
  an absolute cross-call guarantee.
